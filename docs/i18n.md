# Internationalization (i18n)

The site serves two languages — English (`en`) and Chinese (`zh`) — from one codebase,
with English as the fallback. Words come from **three sources**, each with its own
mechanism, and a Chinese reader can flip the whole page from Simplified to Traditional
characters in the browser. This document is the technical reference; the church-facing
overview is [`docs/features/i18n.md`](features/i18n.md).

## Locales & routing

- `src/lib/locales.ts` defines `LOCALES` (`['en', 'zh']`), `DEFAULT_LOCALE` (`en`), and the
  helpers `parseLocale`, `localePath`, `pathWithoutLocale`, and `pickLocaleFromHeader`.
- Public URLs are prefixed by locale: `/en/…` and `/zh/…`, served by the route folder
  `src/pages/[locale]/`.
- A bare `/` is content-negotiated from the `Accept-Language` header in `src/middleware.ts`
  and redirected (302) to the matching localized home page.
- The header language toggle links to the **same page** in the other locale via
  `localePath`, so switching language never loses the reader's place.

## Source 1 — UI dictionaries (fixed labels)

Buttons, menus, and other fixed interface strings live in typed dictionaries:

- `src/i18n/en.ts` and `src/i18n/zh.ts` — the **same key set** in both files.
- `t(locale, key, vars)` looks up a key and interpolates `{vars}`, **HTML-escaping** the
  interpolated values (see [`SECURITY.md`](../SECURITY.md)).

**Parity is enforced by a test.** `test/i18n.test.ts` fails the build if the two
dictionaries disagree — a missing key, a mismatched `{placeholder}`, or an empty value.
You cannot merge a change that adds an English label without its Chinese counterpart.

## Source 2 — Database content (editable, per-language)

Editable content that staff type in — ministry names, event titles, announcements — is
stored in **companion `*_i18n` tables**, one translation row per language, rather than
extra columns on the base row.

A query joins the base table to its `*_i18n` table twice: a `LEFT JOIN` on the **requested**
locale and a `LEFT JOIN` on the **default** locale (`en`), then `COALESCE`s each field so a
missing translation falls back to English instead of rendering blank. One helper builds
these joins (`i18nJoin` in `src/lib/db.ts`). Admin forms render the translatable fields
**side by side**, one group per language, so a translation is entered alongside the
original.

The important property: **adding a language means inserting rows, never adding columns.**
The database shape is fixed no matter how many locales you support.

## Source 3 — Content collections (Markdown prose)

Longer prose pages — "About", "Plan a Visit", beliefs, privacy, the pastor's articles,
fellowship and staff entries — are Markdown files in Astro **content collections**:

```
src/content/{pages,articles,fellowships,staff}/{en,zh}/<slug>.md
```

`src/content.config.ts` defines each collection's frontmatter schema (validated at
`astro check`). `src/lib/content.ts` loads an entry for the requested locale and, when the
translation is missing, **falls back to English and shows a "not translated" badge** rather
than a blank page.

## Simplified → Traditional toggle

Chinese content is authored once, in **Simplified** characters. Readers who prefer
**Traditional** characters flip the whole page with a toggle; the conversion happens **in
the browser**, instantly, and the choice is remembered. No second copy is stored.

- `src/lib/s2t.ts` (`toTraditional`) with the generated table `src/lib/s2t-table.json`
  (built by `scripts/generate-s2t.mjs` from `opencc-data`).
- `src/lib/zh-client.ts` is the small client script that performs the toggle.

## Emails follow the reader

Transactional emails (magic links, scheduling requests, the weekly digest) go out in the
recipient's saved language when known, and **bilingually stacked** when not — so no one
receives a message they cannot read. See [`docs/features/email-automation.md`](features/email-automation.md).

## Add-a-locale checklist

Adding a third language (say, Spanish `es`) touches these places and **never the database
schema**:

1. **Register the locale.** Add `'es'` to `LOCALES` in `src/lib/locales.ts`.
2. **Add a UI dictionary.** Create `src/i18n/es.ts` with the **exact same key set** as
   `en.ts` (the parity test will tell you precisely what is missing). Wire it into the `t()`
   lookup alongside `en`/`zh`.
3. **Add content rows/folders.** Insert `es` rows in the `*_i18n` tables for editable
   content, and create `src/content/{pages,articles,fellowships,staff}/es/…` Markdown files.
   Anything not yet translated falls back to English automatically.
4. **Decide on any script conversion.** The Simplified→Traditional toggle is Chinese-specific;
   a new language needs no equivalent unless you want one.
5. **Verify.** Run the suite — `test/i18n.test.ts`, `test/locales.test.ts`,
   `test/s2t.test.ts` — and check the toggle and fallbacks in `npm run dev`.

## Where it lives

| Thing | Path |
|---|---|
| Locales & routing | `src/lib/locales.ts`, `src/middleware.ts` |
