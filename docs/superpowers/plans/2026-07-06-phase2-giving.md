# Phase 2: Giving Module ŌĆö Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A Supabase-only Giving module: one-time + recurring digital giving via Stripe Checkout, funds admin for the finance team, manual check/cash gift recording, Stripe-FDW reconciliation, and a household giving view.

**Architecture:** Money flows through hosted Stripe Checkout (mode `payment` / `subscription`, ad-hoc `price_data`, funds carried as metadata ŌĆö no Stripe Products). A signature-verified webhook writes `gifts` rows (idempotent via unique Stripe-id indexes). Recurring gifts mirror Stripe subscriptions in `recurring_gifts` and are managed in the hosted Customer Portal. The Supabase Stripe FDW (`stripe.*` foreign tables) powers a reconciliation admin page, feature-detected so plain Postgres degrades gracefully. All business logic lives in `src/lib/*Db.ts` + `src/lib/stripe.ts` (thin Astro pages), so it is testable in the `pg` vitest project without a browser.

**Tech Stack:** Stripe REST API via `fetch` (no SDK), WebCrypto HMAC for webhook signatures, postgres via the Phase 1 `AppDb` adapter.

**Spec:** `docs/superpowers/specs/2026-07-06-supabase-giving-registration-design.md`
**Depends on:** Phase 1 complete (`AppDb`, `PgAdapter`, `locals.db`, module gating with `requiresBackend: 'supabase'`, `test/pg` project).

## Global Constraints

- Giving is Supabase-only: every giving page first checks `Astro.locals.modules.has('giving')` (the middleware already 404s owned routes; in-page checks cover shared surfaces like `/give`).
- Amounts are **integer cents**; input parsing rejects NaN, <100 ($1 min), >9_999_999 ($99,999.99 max). Currency from setting `giving.currency`, default `'usd'`.
- Secrets: `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET` (Wrangler secrets; document in `.dev.vars.example`). Never log either; never call live Stripe in tests (mock `fetch`).
- Guests may give one-time (name + email required); recurring requires sign-in. Signed-in gifts always link `person_id`.
- i18n: every new UI string gets `en` + `zh` keys (parity test enforces).
- All new SQL portable in style (TEXT timestamps via `datetime('now')`, INTEGER 0/1 flags) even though these tables are Postgres-only ŌĆö consistency with the rest of the codebase.
- After every task: `npm test` green; with DATABASE_URL, `npx vitest run --project pg` green; `npm run check` clean. Commit per task.

`verifyStripeWebhook` implementation core (timing-safe via `crypto.subtle.verify`):

```ts
export async function verifyStripeWebhook(
  payload: string,
  sigHeader: string,
  secret: string,
  toleranceSeconds = 300,
  nowSeconds = Math.floor(Date.now() / 1000),
): Promise<Record<string, unknown> | null> {
  const parts = new Map(
    sigHeader.split(',').map((kv) => {
      const i = kv.indexOf('=');
      return [kv.slice(0, i).trim(), kv.slice(i + 1)] as const;
    }),
  );
  const t = parts.get('t');
  const v1 = parts.get('v1');
  if (!t || !v1 || !/^\d+$/.test(t) || !/^[0-9a-f]{64}$/.test(v1)) return null;
  if (Math.abs(nowSeconds - Number(t)) > toleranceSeconds) return null;
  const key = await crypto.subtle.importKey(
    'raw', new TextEncoder().encode(secret), { name: 'HMAC', hash: 'SHA-256' }, false, ['verify'],
  );
  const sigBytes = new Uint8Array(v1.match(/../g)!.map((h) => parseInt(h, 16)));
  const ok = await crypto.subtle.verify('HMAC', key, sigBytes, new TextEncoder().encode(`${t}.${payload}`));
  if (!ok) return null;
  try { return JSON.parse(payload) as Record<string, unknown>; } catch { return null; }
}
```
