# Supabase Backend + Giving + Registration — Design Spec

Date: 2026-07-06
Status: Approved by Leo (session decisions recorded below)

## Goal

1. Add **Supabase (Postgres)** as a second database option, chosen at setup time.
   Every existing module and capability must work identically on either backend.
2. Add two **Supabase-only** modules (unavailable and hidden on D1):
   - **Giving** — one-time and recurring digital giving via Stripe, a funds admin
     for the finance team, manual check/cash gift recording, and a household
     giving view for members.
   - **Registration** — admin-built events with custom question forms and
     optional Stripe payment collection.

## Decisions made with the user

- Payment collection uses **Stripe Checkout (hosted)** + webhooks. The
  [Supabase Stripe FDW](https://supabase.com/docs/guides/database/extensions/wrappers/stripe)
  is read-only for payment objects (write support covers only Customers,
  Products, Subscriptions), so it is used for **reconciliation/reporting SQL**
  in the giving admin, not for collecting payments.
- **Guests can give** (one-time, name + email). Recurring giving requires
  sign-in (subscription management). Signed-in gifts link to person + household.
- **Guests can register** for events; signed-in members get prefilled forms.
- Workers → Supabase connectivity via **Cloudflare Hyperdrive + postgres.js**
  (native SQL, lowest latency, free tier available).
- Currency defaults to **USD** (`giving.currency` setting).
- "Finance team" = a **`people.finance` boolean flag** (managed in People
  admin). Giving admin is visible to `admin` role or finance-flagged people.
  No new role is added.

## Architecture choice

**Chosen: A — D1-shaped adapter layer.** The app has 279 raw
`db.prepare(...).bind(...)` call sites and no ORM. Instead of rewriting them:

- Define an **`AppDb`** structural interface = the subset of `D1Database` the
  app uses (`prepare(sql)` → statement with `bind/first/all/run`, plus
  `batch()`). `D1Database` satisfies it as-is; a new `PgAdapter` implements it
  over postgres.js.
- Meet in the middle on SQL dialect:
  - D1/SQLite supports `RETURNING` → rewrite `meta.last_row_id` and
    `SELECT MAX(id)` sites to `INSERT ... RETURNING id` (portable, removes races).
  - Postgres allows user-defined functions → create **SQLite-compat functions**
    `datetime(text, variadic text[])` and `date(text, variadic text[])` in the
    Postgres schema implementing the modifiers actually used
    (`±N minutes/hours/days/years`, `weekday 0`, `start of day`). The ~120
    `datetime('now', ...)` call sites (including DDL defaults and the seed) run
    unchanged on both backends.

Rejected: B — Drizzle/ORM rewrite (rewrites the whole data layer; massive churn
for no user-visible gain). C — parallel per-backend `*Db.ts` implementations
(two copies of business logic guarantees drift).

## Phase 1 — Supabase backend

### Backend selection

- `wrangler.jsonc` gains a `DB_BACKEND` var (`"d1"` default | `"supabase"`) and
  a commented-out `hyperdrive` binding block (`HYPERDRIVE`).
- New `src/lib/dbProvider.ts` exposes `getDb(env): AppDb` — the single seam.
  D1: returns `env.DB`. Supabase: lazy per-isolate `PgAdapter` over
  `postgres(env.HYPERDRIVE.connectionString)` with reconnect handling.
- Sweep: type annotations `D1Database` → `AppDb` across `src/lib` and pages;
  `vars.DB` access goes through `getDb`. Mechanical, no logic changes.
- `locals.dbBackend` set in middleware for module gating and page hints.

### PgAdapter semantics

- Placeholder translation `?` / `?N` → `$n` with a scanner that skips string
  literals and comments.
- `.first()` / `.all()` / `.run()` return D1-shaped results
  (`{ results, meta: { changes } }`); `run().meta.changes` = affected rows.
- `batch(stmts)` executes all statements inside one `BEGIN/COMMIT` transaction
  (matches D1 batch atomicity, on which admin revision snapshots rely).
- Postgres unique violations (SQLSTATE 23505) recognized by updating
  `isUniqueViolation()` (`adminDb.ts`, `teamDb.ts`) to match both backends'
  messages.

### Portability sweep (works on both backends)

- 13 `meta.last_row_id` reads + 5 `SELECT MAX(id)` last-insert-id patterns →
  `RETURNING id`.
- 3 search queries using SQLite's case-insensitive `LIKE` →
  `LOWER(col) LIKE LOWER(?)` (identical behavior on both).
- 1 `INSERT OR IGNORE` (`giftDb.ts`) → `ON CONFLICT DO NOTHING`.

### Postgres schema & migrations

- New `migrations-supabase/` directory:
  - `0001_init.sql` — compat functions + full port of the ~42-table schema.
    `INTEGER PRIMARY KEY` → `integer GENERATED BY DEFAULT AS IDENTITY` (allows
    explicit-ID seed inserts; runner resets sequences after seeding). Keep
    `TEXT` timestamps (same `YYYY-MM-DD HH:MM:SS` format) and `INTEGER 0/1`
    booleans for cross-backend SQL parity. Avoid `bigint` columns so
    postgres.js returns plain numbers.
  - `0002_giving.sql`, `0003_registration.sql` — new module tables (below).
  - `0004_stripe_fdw.sql.example` — documented, optional FDW setup (wrappers
    extension + Vault key + `stripe` schema foreign tables); run manually in
    the Supabase SQL editor because it needs the church's Stripe key.
- Runner: `scripts/db/migrate-supabase.mjs` (postgres.js, `SUPABASE_DB_URL`
  env, `_migrations` tracking table) + npm scripts `db:migrate:supabase`,
  `db:seed:supabase`.
- Seed: reuse `seed/dev-seed.sql` on Postgres thanks to the compat functions;
  any residual SQLite-only statements get a documented PG override file.

### Unchanged

R2 media, email sending, auth/session, crons all stay as-is. The nightly
D1→R2 backup cron no-ops with a log line on the Supabase backend (Supabase has
its own backups; documented).

## Phase 1.5 — Module gating for backend-specific modules

- `ModuleDef` gains `requiresBackend?: 'supabase'`.
- `MODULE_KEYS` grows to 13: `giving`, `registration` appended.
- On D1, backend-gated modules are force-disabled regardless of settings:
  `getEnabledModules(db, backend)` filters them; middleware 404s their routes.
- Admin Modules panel renders them greyed-out with a "requires the Supabase
  database" note when on D1.
- Route ownership (longest-prefix, same pattern as `gifts` owning
  `/serve/gifts` inside `serve`'s `/my`):
  - `giving`: public `/give/checkout`, `/my/giving`, `/api/giving`;
    admin `/admin/giving`. The existing `/give` page stays core and renders the
    full giving UI only when `locals.modules.has('giving')` (else the current
    external-link behavior).
  - `registration`: public `/register`, `/api/register`; admin
    `/admin/registration`.
- `/api/stripe/webhook` is a **core** route (shared by both modules); handlers
  check module enablement per event.

## Phase 2 — Giving module

### Tables (`0002_giving.sql`, Supabase-only)

- `funds` — `id`, `fund_number TEXT UNIQUE NOT NULL`, `active`, `sort`,
  timestamps; `fund_i18n(fund_id, locale, name)` for bilingual names.
- `gifts` — `id`, `person_id` (nullable FK), `donor_name`, `donor_email`
  (guest gifts), `fund_id` FK, `amount_cents INTEGER`, `currency`,
  `method CHECK ('card','check','cash')`,
  `status CHECK ('pending','succeeded','failed','refunded')`,
  `stripe_checkout_session_id`, `stripe_payment_intent_id`,
  `stripe_subscription_id` (recurring occurrences), `check_number`,
  `received_on` (date; check/cash), `recorded_by` (admin person_id for manual
  entries), `note`, `created_at`. Household attribution is resolved at query
  time via `household_members` (households change; gifts don't move).
- `recurring_gifts` — `id`, `person_id`, `fund_id`, `amount_cents`, `currency`,
  `interval CHECK ('week','month')`, `stripe_subscription_id UNIQUE`,
  `status CHECK ('active','past_due','canceled')`, timestamps. Local mirror for
  display; webhook-synced.

### Flows

- **Public `/give` page** (module on): active fund picker, amount, frequency
  (one-time / weekly / monthly). Guests enter name + email (one-time only);
  signed-in users are auto-linked. Submit → `POST /api/giving/checkout` →
  Worker creates a Stripe Checkout Session (`mode: payment` or `subscription`,
  ad-hoc `price_data` — funds are metadata `fund_id`, no Stripe Products
  needed; `metadata` also carries `person_id`/donor info) → redirect to Stripe.
  Success URL `/[locale]/give/thanks`; the gift row is created by the webhook,
  not the redirect.
- **Recurring management**: "Manage recurring giving" on `/my/giving` opens a
  Stripe **Customer Portal** session (cancel/update there; no in-app CRUD).
- **Webhook `/api/stripe/webhook`**: signature-verified
  (`STRIPE_WEBHOOK_SECRET`). Handles `checkout.session.completed` (one-time
  gift → succeeded; subscription setup → recurring_gifts row),
  `invoice.paid` (each recurring charge → one gift row), `charge.refunded`
  (gift → refunded), `customer.subscription.updated/deleted` (status sync).
  Idempotent by Stripe object IDs.
- **Admin `/admin/giving`** (admin role or `people.finance`): funds CRUD
  (bilingual name + fund number); manual gift entry (person search or
  free-text guest name, fund, amount, method check/cash, check number, date
  received); gifts ledger with fund/date-range filters and totals by fund;
  **reconciliation page** querying Stripe FDW foreign tables
  (`stripe.checkout_sessions`, `stripe.payment_intents`) against local `gifts`
  to flag missing webhooks / amount drift / refunds. Feature-detects the
  `stripe` schema; when absent, shows setup instructions instead.
- **Member view `/my/giving`**: gifts of everyone in the viewer's household
  (digital + check + cash), yearly totals and per-fund breakdown, plus own
  recurring gifts with portal link.

### Stripe config

Worker secrets: `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`. Stripe calls use
plain `fetch` against the Stripe REST API (no SDK dependency; Workers-friendly,
form-encoded bodies).

## Phase 3 — Registration module

### Tables (`0003_registration.sql`, Supabase-only)

- `reg_events` — `id`, `starts_at`, `ends_at`, `location`, `capacity`
  (nullable = unlimited), `price_cents` (nullable/0 = free), `currency`,
  `opens_at`, `closes_at`, `active`, timestamps;
  `reg_event_i18n(event_id, locale, title, description)`.
- `reg_questions` — `id`, `event_id` FK, `sort`,
  `type CHECK ('text','textarea','select','checkbox','yesno')`, `required`,
  `options` (JSON array for select/checkbox);
  `reg_question_i18n(question_id, locale, label)`.
- `registrations` — `id`, `event_id` FK, `person_id` (nullable), `name`,
  `email`, `status CHECK ('pending','confirmed','cancelled')`, `amount_cents`,
  `stripe_checkout_session_id`, `stripe_payment_intent_id`, timestamps.
- `reg_answers` — `registration_id` FK, `question_id` FK, `value TEXT`
  (checkbox answers JSON-encoded).

### Flows

- **Public `/register`**: list of open events (active, within opens/closes,
  not full). `/register/[id]`: event details + dynamic question form; guests
  enter name + email, members prefilled. Free events → `confirmed`
  immediately + confirmation screen. Paid events → `pending` registration +
  Stripe Checkout (`mode: payment`, metadata `registration_id`); webhook
  `checkout.session.completed` → `confirmed`; `checkout.session.expired` →
  `cancelled` (frees the seat).
- **Capacity**: confirmed + pending count against capacity; full → closed
  message. No waitlist in v1.
- **Admin `/admin/registration`** (admin/editor per existing route policy):
  event CRUD with question builder (add/remove/reorder/required/options,
  bilingual labels); registrations list with answers; CSV export; cancel
  registration (refunds handled in the Stripe dashboard in v1).

## Testing & validation

- Existing 490+ tests keep running against D1 (regression floor).
- New Postgres test layer (vitest `node` project, real local Postgres via
  `DATABASE_URL`; skipped when unset; CI adds a `postgres` service container):
  - PgAdapter unit tests: placeholder translation, batch transactionality,
    error mapping, `meta.changes`.
  - Compat-function tests: `datetime`/`date` modifier outputs vs SQLite
    semantics for every modifier form used in the repo.
  - Migration test: `migrations-supabase` applies cleanly from scratch.
  - **Cross-backend parity suite**: representative `*Db` functions for every
    core table run against Postgres (settings, auth tokens, people,
    households, ministries, teams, plans, events, content modules).
- Stretch goal: run the whole e2e suite against Postgres via miniflare's
  hyperdrive binding pointed at local Postgres (`vitest.e2e.pg.config.ts`).
  If miniflare support falls short, the parity suite + `wrangler dev` manual
  smoke is the fallback.
- Stripe is never called live in tests: `fetch` is mocked; webhook tests
  construct real signatures with a test secret.
- Giving/Registration each ship unit + e2e tests and screenshots.
- FDW reconciliation code is feature-detected and covered by tests with a
  stubbed `stripe` schema (plain tables mimicking the foreign tables), since
  the wrappers extension only exists on Supabase.

## Phase 4 — Setup & docs

- `docs/supabase-setup.md`: create Supabase project → connection string →
  `wrangler hyperdrive create` → paste id + `DB_BACKEND=supabase` →
  `npm run db:migrate:supabase` → secrets → deploy → first admin via Supabase
  SQL editor → Stripe dashboard steps (API key, webhook endpoint, Customer
  Portal) → optional Stripe FDW setup for reconciliation.
- `docs/deploy.md` + `docs/cloudflare-setup.md` + README gain the
  "choose your database" fork.
- Module docs: `docs/features/giving.md`, `docs/features/registration.md`
  (non-technical English, per project convention), module rows in README.
- Demo seed rows for funds/gifts/reg events on the Supabase path.

## Execution workflow

Fable (this session) planned and designed; **Opus subagents** execute the
implementation plan phase by phase (each task has verification criteria and
screenshots for UI work); Fable validates at the end: full unit suite,
`astro check`, build, D1 e2e, Postgres parity/e2e, and a real-flow walkthrough
of giving and registration on local Postgres.

## Out of scope (v1)

Waitlists, in-app refunds, giving statements/tax receipts (PDF), ACH/bank
debit, per-event member-only registration toggle (guests allowed everywhere),
donor-covers-fees option, pledge campaigns, non-USD multi-currency.
